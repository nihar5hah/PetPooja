Hackathon Final Submission

PetPooja

An AI restaurant ordering and revenue intelligence platform that connects voice ordering, menu validation, combo recommendations, transactional persistence, and live business analytics.

1. Executive Summary

PetPooja is designed to solve a real restaurant operations problem: voice ordering, upsell intelligence, and business analytics usually live in disconnected systems. Orders may be captured manually or through a voice layer, but they often do not feed back into the restaurant's revenue intelligence loop in real time.

PetPooja connects these pieces into one workflow. A voice agent can validate spoken menu items against a structured dataset, recommend relevant combos from historical POS transaction patterns, create and persist the final order, and immediately update the analytics layer powering the dashboard. This means every confirmed order becomes both an operational event and an analytics event.

The system is implemented using a FastAPI backend, a Next.js dashboard, Supabase PostgreSQL with Realtime, Retell for the voice agent, and ngrok for development-time public exposure.

2. Problem Statement

Restaurants commonly face three operational gaps:

  1. Orders taken through conversation can contain transcription or menu matching errors.
  2. Upsells are often based on staff habit rather than data-backed purchase patterns.
  3. Business dashboards do not reflect the impact of newly placed orders immediately.

In voice-ordering scenarios, this problem gets worse. Spoken item names may be imprecise, customers may order several items in a single sentence, and agents need to keep the interaction natural while still being accurate. At the same time, restaurant owners want every order to improve visibility into revenue, popularity, and combo behavior.

PetPooja addresses this by combining voice-safe APIs, structured order persistence, analytics recomputation, and dashboard synchronization into one system.

3. Solution Overview

Conversation Layer

Retell handles the customer-facing voice interaction and uses backend functions for validation, combo lookup, and order finalization.

Application Layer

FastAPI exposes endpoints for menu retrieval, item validation, combo recommendations, voice webhooks, order creation, analytics, and dashboard data.

Data & Intelligence Layer

Supabase PostgreSQL stores orders, order items, transactions, metrics, combo rules, voice logs, and failure queues.

Presentation Layer

Next.js presents KPIs, menu engineering, combo insights, pricing suggestions, strategy cards, and order monitoring with realtime refresh.

4. System Architecture

Customer Call | v Retell Voice Agent | | function calls / webhook v FastAPI Backend |-- /menu |-- /validate-item |-- /combo-recommendations |-- /create-order |-- /voice/retell-webhook |-- /dashboard/* |-- /extended-analytics/* v Supabase PostgreSQL |-- orders |-- order_items |-- pos_transactions |-- menu_metrics |-- combo_rules |-- voice_call_logs |-- failed_order_queue v Next.js Dashboard | v Realtime KPI + order + combo updates

The architecture deliberately keeps the integration path simple: the voice agent never writes directly to the database. All order-related decisions flow through the backend, which centralizes validation, recommendation logic, persistence, and analytics synchronization.

5. Core Features

5.1 Menu Validation

The backend validates customer-requested items using the active menu dataset. It supports alias-based and fuzzy matching to handle transcription errors while still preventing invalid orders.

5.2 Combo Recommendations

Combo suggestions are generated from historical transaction behavior. The system mines association rules using support, confidence, lift, and derived combo scores, then exposes a simplified voice-safe recommendation format for the Retell agent.

5.3 Structured Order Creation

On confirmation, the backend creates an order header, line items, and mirrored analytics transaction rows. This ensures the same confirmed order is visible both operationally and analytically.

5.4 Automatic Analytics Refresh

After a confirmed order is stored, menu metrics and combo rules are recomputed. Supabase Realtime then broadcasts changes so the dashboard can refresh automatically.

5.5 Dashboard Intelligence

The dashboard provides visibility into total orders, revenue, profit, menu engineering classes, combo performance, price opportunities, hidden gems, and watch-list items.

6. End-to-End Workflow

  1. A customer calls the voice assistant.
  2. The assistant extracts the requested items and quantities.
  3. Each item is validated against the menu dataset.
  4. The assistant requests combo recommendations for a validated main item.
  5. The customer accepts or declines suggested add-ons.
  6. The assistant summarizes the final order and asks for confirmation.
  7. On confirmation, the backend creates the order and order items.
  8. The backend inserts transaction rows into the analytics dataset.
  9. Analytics recompute and database change events are published.
  10. The dashboard refreshes and reflects the newly placed order.
The most important design choice is that confirmed voice orders are written into both operational tables and analytics fact tables. This removes the lag between ordering and business visibility.

7. Technical Implementation

Backend Stack

  • FastAPI 0.115
  • SQLAlchemy 2.0
  • PostgreSQL via psycopg 3
  • pandas for dataset and metric processing
  • mlxtend for association rule mining
  • Pydantic 2 for API schemas

Frontend Stack

  • Next.js 14 and React 18
  • Tailwind CSS 4
  • Framer Motion for UI motion
  • Recharts for analytics visualization
  • Supabase JS client for realtime subscriptions

Data Sources

  • Menu_Items_Set2.csv: menu metadata, pricing, food cost, margin, category, item status
  • POS_Transactions_Set2.csv: historical order lines used for combo mining and menu analytics

8. API and Integration Summary

Endpoint Method Purpose
/menu GET Returns active menu items for voice-agent use.
/validate-item POST Validates a spoken item against the menu dataset.
/combo-recommendations GET Returns detailed or simplified combo suggestions.
/create-order POST Creates a confirmed order and syncs analytics data.
/voice/retell-webhook POST Handles webhook-driven voice-order flows.
/dashboard/* GET Provides dashboard summaries, menu engineering, combos, orders, logs, and failures.
/extended-analytics/* GET Provides deeper KPI, pricing, basket, and strategy views.

9. Database and Data Model

Table Role
orders Stores confirmed order headers.
order_items Stores line items for each order.
pos_transactions Stores line-level transactional facts used by analytics.
menu_metrics Stores derived menu engineering metrics.
combo_rules Stores mined item association rules for recommendation logic.
voice_call_logs Stores voice interaction traces and payloads.
failed_order_queue Stores failed persistence attempts for retry.
menu_aliases Stores alias mappings to improve spoken item resolution.

13. Challenges and Tradeoffs

  • Realtime analytics recompute inside the order path improves demo visibility but is less scalable than an async job queue.
  • ngrok is excellent for hackathon development but introduces temporary public URLs and manual reconfiguration.
  • Fuzzy item matching improves voice usability but requires thresholds to avoid incorrect substitutions.
  • Writing confirmed orders into analytics tables simplifies live insight updates but couples operational writes with analytics freshness.

14. Future Improvements

  1. Move recompute workloads to background processing.
  2. Add production-grade authentication and role-based access control.
  3. Support multi-restaurant tenancy more explicitly.
  4. Add richer voice-session analytics and QA views.
  5. Improve failure queue observability and retries.
  6. Replace development-only ngrok exposure with stable production deployment.

15. Closing Summary

PetPooja demonstrates a full-stack AI ordering workflow in which a customer conversation becomes a validated order, a business transaction, and a live analytics signal. Its strength is not only in voice interaction or dashboarding alone, but in how the system connects both sides into a single revenue intelligence loop.
